# 烟厂采购标书 Agent 填写系统 — 需求文档

> 版本：v4.0 | 日期：2026-06-04 | 状态：重构

---

## 一、核心目标

上传招标文件（PDF）+ 固定格式的投标模板（Word），由 AI Agent 自动将 PDF 中的信息填写到 Word 模板中，**模板格式零破坏**。

**一句话价值**：PDF 当"资料库"，Word 当"答题卡"，Agent 当"填写员"。

---

## 二、系统概述

### 2.1 系统定位

- **使用场景**：烟厂（烟草行业）采购项目的投标文件编制
- **用户角色**：投标部门人员、项目经理、商务人员
- **运行环境**：浏览器端 + 本地/内网后端服务（烟草行业数据安全要求）

### 2.2 输入输出

| 项目 | 说明 |
|------|------|
| **输入** | ① 招标文件（PDF，20-50页，中文）— 作为参考资料<br>② 投标模板（DOCX，固定格式）— 格式不可变<br>③ 公司信息（预置配置，含资质、业绩等） |
| **输出** | ① 填写完成的投标文件（DOCX），格式与原模板完全一致<br>② 无法自动填写的字段标注【待人工补充】<br>③ 填写完整性报告 |

### 2.3 核心约束

- **格式铁律**：Word 模板的格式、结构、样式**绝对不能被破坏**
- **不编造原则**：PDF 中没有的信息一律标注【待人工补充】，绝不虚构
- Agent 自动填写 + 用户可手动微调
- 支持调用国产大模型 API（OpenAI 兼容格式）
- 文档数据不出域，全程本地/内网处理

## 四、功能模块

### 4.1 模块一：PDF 解析（资料提取）

**目标**：从招标文件 PDF 中提取结构化信息，作为 Agent 填写的"资料库"。

**提取清单**：

| 编号 | 提取项 | 说明 | 示例 |
|------|--------|------|------|
| P-01 | 项目基本信息 | 项目名称、编号、采购单位、代理机构、截止时间、开标时间/地点 | "项目编号：YC-2026-XXX" |
| P-02 | ★号废标条款 | 所有带★号的强制性要求，**一票否决**项 | "★投标人须具备ISO9001认证" |
| P-03 | 技术参数清单 | 货物/服务的技术规格、参数要求 | "功率：≥500W"、"材质：304不锈钢" |
| P-04 | 评分标准 | 评分细则、分值分布、评分维度 | "技术方案：40分"、"报价：30分" |
| P-05 | 格式要求 | 投标文件的排版格式规定 | "正文字体：小四仿宋" |
| P-06 | 附件清单 | 需要提交的附件、证明材料列表 | "营业执照副本复印件" |
| P-07 | 资格条件 | 投标人资格要求 | "注册资金≥500万" |
| P-08 | 商务条款 | 付款方式、交货期、质保期等 | "质保期：3年" |
| P-09 | 投标文件组成 | 投标文件应包含的章节及顺序 | "第一部分：商务部分..." |

**技术实现**：
- 使用 `pdfplumber` 提取 PDF 全文
- 调用 LLM 进行结构化拆解（OpenAI 兼容格式）
- 输出 JSON 格式，存入会话上下文

**验收标准**：
- ★号条款提取率 = 100%（零遗漏，否则可能导致废标）
- 项目基本信息字段完整率 ≥ 95%
- 评分标准提取准确率 ≥ 90%

---

### 4.2 模块二：Word 模板分析

**目标**：理解 Word 模板的结构，识别"哪些位置需要填写"。

**分析内容**：

| 分析项 | 说明 |
|--------|------|
| 占位符识别 | 找到 `{{xxx}}`、`【xxx】`、`____` 等待填写位置 |
| 表格结构 | 识别表格的列名、行数、哪些单元格需要填写 |
| 章节结构 | 识别标题层级、章节顺序 |
| 格式信息 | 记录各位置的字体、字号、对齐等格式（填写时保持一致） |

**技术实现**：
- SuperDoc SDK 服务端打开模板
- `superdoc_get_content` 读取文档内容
- LLM 分析模板结构，生成"填写清单"

---

### 4.3 模块三：Agent 循环填写

**目标**：根据 PDF 资料，自动填写 Word 模板中的待填位置。

**Agent 循环流程**（ReAct 模式）：

```
┌─────────────────────────────────────────────────────┐
│              Agent 循环                               │
│                                                      │
│  ① 读取模板内容                                      │
│       │  superdoc_get_content → 理解文档结构          │
│       ▼                                              │
│  ② 分析需要填写的位置                                 │
│       │  识别占位符、空表格、待填字段                   │
│       ▼                                              │
│  ③ 从 PDF 资料中匹配信息                              │
│       │  LLM 推理：这个位置应该填什么？                │
│       ▼                                              │
│  ④ 调用 SuperDoc SDK 填写                            │
│       │  superdoc_edit → 插入/替换内容                │
│       │  每次填写后保存 → 发送 doc_checkpoint          │
│       ▼                                              │
│  ⑤ 自查                                               │
│       │  完整性：所有位置都填了吗？                     │
│       │  准确性：填的内容和 PDF 一致吗？                │
│       │  格式：模板格式破坏了吗？                      │
│       ▼                                              │
│  ⑥ 通过？                                            │
│       │                                              │
│   ┌───┴───┐                                          │
│   │ 不通过  │ → ⑦ 修正 → 回到 ⑤                       │
│   └───┬───┘                                          │
│   ┌───┴───┐                                          │
│   │  通过   │ → ⑧ 进入下一个填写位置                    │
│   └───────┘                                          │
│                                                      │
│  最大迭代轮数：30轮（参考 lab-report-agent）            │
└─────────────────────────────────────────────────────┘
```

**SuperDoc SDK 工具**：

| 工具名 | 功能 | 是否修改文档 |
|--------|------|-------------|
| `superdoc_get_content` | 读取文档内容 | ❌ 只读 |
| `superdoc_edit` | 插入/替换/删除内容 | ✅ 修改 |
| `superdoc_format` | 格式化文本 | ✅ 修改 |
| `superdoc_mutations` | 批量复合操作 | ✅ 修改 |
| `superdoc_track_changes` | 修订追踪 | ✅ 修改 |

**防护机制**（参考 lab-report-agent）：
- **重复调用检测**：同一工具+参数连续调用3次自动停止
- **软/硬轮次限制**：12轮注入收敛提示，30轮强制停止
- **取消支持**：用户可随时打断 Agent
- **格式保护**：填写时自动继承目标位置的格式

**验收标准**：
- 自动填写率 ≥ 80%（可填写位置中）
- 填写准确率 ≥ 90%（与 PDF 资料一致）
- 格式破坏率 = 0%（模板格式完全保持）
- 单次填写流程 ≤ 5分钟

---

### 4.4 模块四：实时预览与手动编辑

**目标**：用户可实时查看 Agent 填写过程，并手动微调。

**功能要求**：

| 编号 | 功能 | 说明 |
|------|------|------|
| E-01 | 实时预览 | Agent 每次填写后，右栏自动刷新显示最新文档 |
| E-02 | 手动编辑 | 用户可在 SuperDoc 编辑器中直接修改内容 |
| E-03 | 工具栏固定 | 编辑器工具栏滚动时固定在顶部 |
| E-04 | 导出下载 | 导出填写完成的 DOCX 文件 |

**技术实现**：
- 前端 `<SuperDocEditor>` 组件渲染文档
- Agent 每次 `superdoc_edit` 后保存文档，发送 `doc_checkpoint` SSE 事件
- 前端收到事件后更新 `docUrl`，编辑器重新加载
- 工具栏使用 `position: sticky` 固定

---

### 4.5 模块五：填写完整性报告

**目标**：自动检查填写完整性，输出报告。

**检查项**：

| 编号 | 检查项 | 说明 |
|------|--------|------|
| C-01 | 填写覆盖率 | 已填写位置 / 总待填位置 |
| C-02 | 信息准确性 | 填写内容与 PDF 资料是否一致 |
| C-03 | 格式完整性 | 模板格式是否被破坏 |
| C-04 | ★条款覆盖 | 所有★条款是否已响应 |
| C-05 | 待补充项 | 标注【待人工补充】的位置列表 |

## 五、公司信息预置

### 5.1 配置内容

公司信息作为系统的"知识库"，在 Agent 填写时自动匹配调用。

| 类别 | 字段 |
|------|------|
| **基本信息** | 公司全称、简称、统一社会信用代码、法定代表人、注册地址、联系人、电话、邮箱 |
| **资质证书** | 营业执照、ISO认证、行业许可证、专利证书等（含有效期） |
| **业绩案例** | 历史合同（项目名称、金额、时间、甲方） |
| **产品参数** | 公司产品的技术参数库（用于与招标参数匹配） |
| **人员信息** | 项目经理、技术人员、售后人员的资质信息 |

### 5.2 配置格式

- 主配置文件：`config/company_info.json`（结构化数据）
- 资质文件：`config/certificates/` 目录下的扫描件
